Omicall STT · Data pipeline

Mỗi cuộc gọi được xử lý Speech-to-Text, lưu 3 tầng:

  • Audio file (.mp3) — bản gốc, lưu vĩnh viễn
  • Transcript text — toàn bộ nội dung hội thoại dạng văn bản
  • Tóm tắt AI — Claude tóm tắt: điểm chính, cam kết, việc cần làm tiếp theo

call_id · occurred_at · owner_id · deal_id
audio_url · transcript_text · ai_summary · sentiment_score

🗂 Dữ liệu thô phần mềm (JSON/raw)
BaseCRM · Zalo · Email · n8n Airbyte

Toàn bộ dữ liệu từ phần mềm lưu nguyên bản trước khi chuẩn hóa:

  • Sự kiện deal — mọi thay đổi giai đoạn, ghi chú, file đính kèm từ BaseCRM
  • Lịch sử liên hệ — thông tin KH, lịch sử cập nhật
  • Tin nhắn chat — Zalo OA, Zalo cá nhân, Facebook

Đối tượng: Data engineer (audit) · AI agent (UC17 hỏi đáp lịch sử)

⚠️
Nguyên tắc bắt buộc khi xây Data Mart
✕ Không đọc thẳng từ Data Lake — luôn build qua DWH, tránh query raw JSON tốn kém và không nhất quán
✕ Không để Claude query DWH trực tiếp — phải qua data mart đã pre-join, đảm bảo latency < 3 giây
✕ Không bỏ Row-level Security — mỗi mart phải có RLS theo đúng đối tượng, sales không được xem data của nhau
✕ Không rebuild toàn bộ — chỉ incremental refresh records mới, tránh vượt budget BigQuery 10M/năm

BƯỚC 6 — Trả kết quả cho người dùng
Claude nhận JSON từ BigQuery → tổng hợp thành ngôn ngữ tự nhiên tiếng Việt → kèm số liệu cụ thể, cảnh báo nếu phát hiện vấn đề, gợi ý hành động tiếp theo. Latency toàn bộ: 3–5 giây.
Lý doTại sao Claude kết nối Data Mart thay vì Data Warehouse?
✕ Nếu Claude query thẳng Data Warehouse
⚠ Chậm: DWH chứa hàng triệu raw records — mỗi query phải join nhiều bảng, latency 10–30 giây
⚠ Tốn kém: BigQuery tính tiền theo dữ liệu scan — query DWH lớn có thể vượt budget 10M/năm
⚠ Rủi ro bảo mật: DWH chứa mọi thứ — khó kiểm soát Claude có thể xem gì, ai được xem gì
⚠ Dễ sai: Claude phải tự join nhiều bảng → có thể sinh SQL sai logic nghiệp vụ
✓ Khi Claude query qua Data Mart
✓ Nhanh: Mart đã pre-aggregated, pre-joined — query đơn giản, latency < 3 giây
✓ Tiết kiệm: Mart nhỏ gọn hơn DWH 10–50 lần — chi phí scan thấp, nằm trong budget
✓ Kiểm soát bảo mật: Mỗi mart có RLS riêng — sales chỉ thấy deal của mình, lãnh đạo thấy tất cả
✓ Chính xác hơn: Logic join và tính toán đã được định nghĩa sẵn trong mart — Claude chỉ cần SELECT
Ví dụ thực tế: Câu hỏi "Deal nào của team Nam > 500M sắp hết hạn?"
Query vào mart_revenue (đã pre-join deals + accounts + owners, đã lọc theo RLS) → 0.2MB data scan → < 2 giây
Nếu query DWH raw → phải join fact_deals × dim_accounts × dim_owners × dim_dates → 2GB+ data scan → 15–30 giây + tốn ~0.01$ mỗi query
